# 二手书双账号测试发布记录（2026-05）

## 目的

验证 `xiaohu-wechat-publishing` 在**非 AI/Agent 主题**下是否仍能稳定完成：

- 双账号差异化写作
- Swiss Minimal 封面渲染
- Markdown 常见元素排版
- 无正文图场景发布
- AI 披露拦截不误伤

## 测试主题

同一主题，不同文风：**年轻人为什么又开始买二手书**

### 熵增时刻
- 标题：`年轻人为什么重新迷上二手实体书？`
- 风格：产业/消费心理分析，克制、结构化

### 思想的野路子丶
- 标题：`年轻人又开始买二手书了，但他们买的根本不是书`
- 风格：锐利、观点驱动、局外人观察

## 刻意覆盖的 Markdown 元素

两篇稿子都主动覆盖：

- GFM 表格
- fenced code block
- blockquote
- `[!tip]` 提示容器
- 有序列表
- 无序列表
- 行内代码
- 分隔线 `---`

经验：如果是在“测 skill”，不要只写一篇能发的稿；要故意把常见元素都打到一遍，才知道排版系统是不是稳定。

## 关键结论

1. **<2500 字完全可以 0 张正文图**。这次两篇都在 1600 字左右，靠表格/引用/列表/代码块完成信息分层，0 图照样是合格成品。
2. **“表格能解决的，不画图”这条规则非常关键**。比起硬插正文图，排版元素本身更适合承担信息组织任务。
3. `publish_pipe.py` 当前日志里这句可能误导：

```text
上传正文图片 (2 本地 + 0 外部)...
上传完成: 0 成功, 0 失败
```

文章实际没有正文图。说明日志统计口径和真实上传对象之间存在错位；未来排查时优先看 `HTML 图片数` 和最终 `<img>`。

## 对未来 agent 的提醒

- 测试 skill 时，优先选一个**和最近主任务完全无关**的话题，避免“因为熟悉主题所以看起来没问题”
- 测试稿不必追求正文图；如果是短文，优先测试排版元素是否覆盖完整
- 真实发布前先做 dry-run，确认：AI guard、封面自动识别、HTML 图片数、账号切换
