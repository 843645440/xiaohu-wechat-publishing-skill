# 正文图设计规则（2026-05）

这份文件补充 `xiaohu-wechat-publishing` 对正文图的硬规则，来源于真实测试与用户纠偏。

## 一、何时配图

只在四类场景配图：

1. 数据 / 数字
2. 对比
3. 流程 / 演化
4. 结构关系

如果表格、列表、正文段落已经能更直观地表达，就**不要画图**。

配图数量：
- `<2500 字`：`0–1 张`
- `≥2500 字`：`0–2 张`
- **永远允许为 0**

## 二、正文图不是封面

正文图不是海报、不是 PPT 封面、不是杂志跨页。

默认只保留三类信息：
- 主标题
- 主图形
- 数值标注

下面这些元素默认删除：
- kicker / 栏目眉（如 `DATA · XXX`）
- 数据来源
- 截止日期
- 反转金句
- 角标
- 装饰线
- 无必要的副标注

这些内容如果确实要出现，放回正文段落里。

## 三、生成前先写“表达策略”

未来 agent 做正文图时，先写一句策略，再动手：

格式：

`主题 → 图型分类（数据/对比/流程/结构） → 图形选择 → 唯一判断`

示例：

- `黄金金价对比 → 数据/对比 → 横向条形图 → 当前价已经站上哪个历史台阶`
- `英伟达中国份额变化 → 趋势/演化 → 单线趋势图 → 从巅峰到现在掉了多少`

这样做的作用：
- 防止 agent 一上来就堆设计元素
- 防止把本该用表格表达的内容误画成图
- 强制每张图只服务一个判断

## 四、判断清单

出图前自问：

1. 这张图是哪一类：数据 / 对比 / 流程 / 结构？
2. 这张图读完后，读者脑子里只应留下哪一句话？
3. 为什么不能直接用表格 / 列表 / 正文段落？
4. 有没有把 kicker / 来源 / 反转金句 / 装饰线先删掉？
5. 遮住任何一个元素，主判断还能读懂吗？能，则该元素应该删

## 五、典型反例

### 反例 1：正文图塞满栏目眉、日期、数据源

用户反馈：
- `“市场用脚投票,但star≠终局。数据·GitHubAPI 截至2026-05-16 和最上面的那个GitHub star+时间的这些数据都没必要，尽量简单直观。”`

结论：
- 这些内容应回到正文段落，而不是留在图里

### 反例 2：表格能表达的内容误画成图

用户反馈：
- A vs B 的信息如果用 markdown 表格已经清楚，就不要再画一张密密麻麻的并列图

结论：
- 图是省脑力的，不是让页面看起来更“专业”的
