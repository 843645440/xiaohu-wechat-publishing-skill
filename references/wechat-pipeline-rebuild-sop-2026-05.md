# 公众号双账号自动化 SOP（重构版，2026-05）

> 用途：为“写作 → 配图 → 排版 → 发布”整条双账号公众号流水线提供一份更稳、更快、更不容易出错的人类可读总流程。
> 
> 这不是最终脚本说明，而是重构管线和编写自动校验器时的设计输入。

## 0. 重构原则（新增）
- 封面：轻量参数化，少变量，强模板。
- 正文图：模块化拼装，按文章类型选模块，不做万能大模板。
- 排版：只做格式转换，不负责猜图位。
- 发布：只做最后投递，不负责修正文案。
- 兼容路径：仅保留最少必要兼容，不做默认入口。

## 1. 流程目标

把一个选题、口播、链接或现成稿件，快速稳定地变成：
- 1 篇适合「熵增时刻」的成稿
- 1 篇适合「野路子」的成稿
- 共享一套正文图（默认）
- 各自一张封面
- 最终进入两个公众号草稿箱

优化目标：
1. 更少返工
2. 更少串号 / 串图 / 串标题
3. 更短总耗时
4. 发布前自动拦住半成品

---

## 2. 推荐的新总流程：三段式

### 第一段：输入定骨架
先做最容易影响后续返工的事：
- 抽核心结论
- 抽可用事实
- 标出风险数据
- 定双账号角度
- 定正式标题
- 定正文结构

输出物：
- 双账号标题
- 双账号大纲
- 可用事实清单 / 待确认点清单

**原则：先定标题和结构，再写全文。**

---

### 第二段：生产并行化
把原来串行的任务拆成并行：

并行做三类事：
1. 熵增正文
2. 野路子正文
3. 正文图（共用一套）

然后再分别做：
4. 熵增封面
5. 野路子封面

**默认策略：正文图共用，封面分离。**

这样做的好处：
- 节省重复生图时间
- 保留双账号人格差异
- 减少“图做了两遍但信息一样”的浪费

---

### 第三段：发布前硬校验
在进入发布前，统一做一轮总控检查，只要有一项失败就停止：
- 标题对不对
- 字数够不够
- 图位对不对
- HTML 有没有图
- 封面存不存在
- 最终目录对不对
- 发布账号对不对

这一步的定位不是“可选检查”，而是整个流程最重要的闸门。

---

## 3. 模块化 SOP

### A. 输入阶段
输入源可能是：
- 口播转写
- 新闻链接
- 公众号原文
- 选题一句话
- 用户自写草稿

此阶段必须产出：
- 核心判断一句话
- 双账号角度差异一句话
- 正式标题
- 3～5 节结构

**不建议**：拿到素材后直接开写全文。

---

### B. 标题阶段
这是最容易引发全链路错误的地方。

强制规则：
- Markdown 第一个 `#` 必须是正式标题
- 账号名不能出现在第一个 `#`
- 账号名只能放在文件名、目录名、元数据或注释中
- 写完正文后，必须单独校验一次首个 `#`

推荐自动检查：
- 若第一个 `#` 是“熵增时刻 / 野路子”之类账号名，直接报错退出

---

### C. 写作阶段
双稿同时生产：
- 熵增：更偏产业结构、趋势、科技与资本支撑
- 野路子：更偏牌桌、冲突、秩序感、资本博弈

硬要求：
- 用户要求 2000～3000 字时，默认按目标区间直接写
- 不要先写成 900～1000 字短稿再临时补水
- 正文内应提前预留图位

推荐做法：
- 在正文草稿里就写入 `<!-- img:hero.png -->` 或 `![图](images/hero.png)`
- 不要把插图这件事留到排版后再猜

---

### D. 核验阶段
优先核验：
- 人物名
- 公司名
- 市值量级
- 利润 / 营收 / CapEx
- 时间与增速

处理策略：
- 确认过的，正常写
- 不完全确定的，用“约”“量级”“公开口径”表达
- 不要把近似值写成精确 tick 数据

目标：既保留产业体感，又降低硬伤风险。

---

### E. 正文图阶段
默认只做一套正文图，供双账号共用。

推荐把正文图拆成 **三种模块**：
1. **hero**：总量级冲击 / 文章第一锚点
2. **people**：人物-公司-市值卡 / 盘点型信息
3. **structure**：结构影响图 / 权力地图 / 关系判断

必要时再加一个可选模块：
4. **compare**：对照判断图 / 表面 vs 本质

模块原则：
- 每个模块只解决一个问题
- 每个模块只保留最小必需变量
- 文章先决定要哪几个模块，再决定怎么排
- 不再用一个重型模板去硬吃所有场景

硬规则：
- 图必须插在对应段落后
- 禁止统一堆到文末
- auto 注图不能作为新稿唯一主路径

本轮实测验证出的结论：
- 只靠“按 h2 自动插图”很容易错位
- h2 数量不足时，图会被追加到文末
- 图片没提前复制进目标目录时，注图会直接失败

因此推荐：
1. 写作阶段就预埋图位
2. 正文图先按模块生成，再组合成最终图序列
3. 图片先进入目标目录
4. 再做排版
5. 最后只把注图脚本当验证或轻修工具

---

### F. 封面阶段
封面默认一号一张。

规则：
- 熵增和野路子共用一个视觉系统，但分别出图
- 标题必须是封面的视觉入口
- 封面文件必须真实落到最终目录

硬规则：
- `images/cover.png` 不存在就不能发布
- 发布命令必须显式传 `--cover`
- 不允许静默用正文第一张图当封面

---

### G. 排版阶段
推荐顺序：
1. 确认正文图与封面图都已存在
2. 把图片复制到最终发布目录的 `images/`
3. 再运行 `format.py`
4. 再检查 HTML

重点检查：
- `article.html` 是否存在
- 是否真的有 `<img>`
- 标题是否正确
- 图位是否仍在正确位置
- 输出目录是否是最终发布目录

本轮实测说明：
- 只检查 preview 不够
- 必须检查真实发布 HTML 的标题和 `<img>` 数量

---

### H. 发布阶段
发布前必须全部通过：
- 标题正确
- 字数达标
- HTML 中有图
- 封面存在
- `--account` 正确
- 最终目录正确

推荐命令原则：
- 统一用 `--account` 切号
- 统一用 `--cover` 显式指定封面
- 缺文件即失败，不兜底

---

## 4. 建议新增的总控校验层

建议未来增加一个 `pipeline-checker` 统一检查：
- 首个 `#` 是否为真实标题
- 字数是否达标
- HTML 是否含 `<img>`
- 图片是否不是全部堆在文末
- `images/cover.png` 是否存在
- 发布目录是否正确
- 目标账号是否正确

这个层一旦实现，建议它成为正式发布前的唯一必经入口。

---

## 5. 失败即停规则

以下任一失败，立即停止，不进入发布：
- 标题错误
- 首个 `#` 被账号名占用
- 字数不足
- HTML 无图
- 正文图堆文末
- 封面不存在
- 输出目录错误
- 发布账号错误

---

## 6. 本轮测试暴露的典型失败模式

1. 封面渲染命令失败，但如果不立刻校验文件是否存在，问题会拖到发布阶段才暴露
2. auto 注图在 h2 数不足时会把图片全部 append 到文末
3. 目标目录中没有图片时，注图脚本会报“未找到任何图片”
4. 只检查 preview 页面不足以保证发布质量
5. 发布成功不代表正文图正确，也不代表封面正确
6. 自动改写可能生成 900～1000 字短稿，需要字数闸门拦截

---

## 7. 推荐的最终稳定版流程

1. 输入素材
2. 提炼事实骨架
3. 产出双账号标题与结构
4. 双稿并行写作
5. 共享正文图 + 双封面生成
6. 在正文对应段落落图
7. 准备最终目录
8. 排版生成 HTML
9. 运行总控校验
10. 双账号发布到草稿箱

---

## 8. 适合进一步工程化的方向

如果继续重构，最优先做三件事：
1. 新增 `pipeline-checker` 脚本
2. 把“标题校验 + 字数校验 + 封面存在校验 + HTML 图校验”做成统一命令
3. 把双账号流程变成单命令总入口，而不是人工串脚本
