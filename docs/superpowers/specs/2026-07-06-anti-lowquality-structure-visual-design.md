# 反低创作度结构与视觉去重设计

> 日期：2026-07-06
> 范围：`xiaohu-wechat-publishing` skill、早晚 cron 提示词、发布前检查与发布历史记录。

## 目标

微信低创作度规则关注同一账号近期内容的整体重复感：主题、标题、正文框架、封面、图片和信息增量。当前流程已经约束了单源洗稿、标题党和高风险角度，但把“高质量写作”固化成固定正文结构和固定封面模板，仍可能被识别为框架一致的批量内容。

本次改造目标是保留信息增量要求，同时拆掉固定骨架：

- 同账号近 7 天不连续复用正文结构、标题句式、开头方式、结尾方式、封面原型和正文图类型。
- 每篇先选择适合题材的结构原型，再组织信息组件，不按固定顺序填空。
- 封面从“单模板换字”改为“品牌统一下的多原型轮换”。
- 发布历史记录结构与视觉指纹，发布前对同账号近期历史做相似度告警。

## 设计原则

1. 信息组件固定，呈现顺序不固定。每篇仍必须有事实、多源、原创分析、读者价值和风险边界，但不再规定“事实底座 → 背景解释 → 原创分析 → 普通人影响 → 中国变量 → 结尾”的段落顺序。
2. 品牌系统统一，封面构图不固定。账号可以保持字体、色彩和气质一致，但不能短期连续使用同一封面版式和标题排布。
3. 代码做兜底，不替代编辑判断。发布脚本先给 warning，不默认 block，避免误伤已有人工精品稿；cron prompt 要求命中高相似时重写或换结构。
4. 去重维度按账号隔离。重点比较同一账号近期文章，不把 `xiaocong` 和 `yeluzi` 的差异化内容互相误判。

## 创作规则改造

`quality-and-risk.md`、`SKILL.md` 和两个 cron prompt 都要从“固定结构”改为“结构菜单”。

正文结构原型至少包括：

- 事件拆解型：事件事实 → 关键变化 → 影响对象 → 后续观察。
- 案例切入型：一个具体场景/人物 → 背后规则 → 更大变化 → 读者建议。
- 对比型：过去/现在、表面/本质、平台/用户等对照展开。
- 问答型：用 4-6 个真实读者问题组织全文。
- 时间线型：按节点展示变化，不用普通分析稿顺序。
- 清单型：3-5 个变化、误区或行动建议，每条必须有事实支撑。
- 反常识型：常见理解 → 事实反证 → 真正原因 → 影响。
- 现场观察型：从具体生活/工作场景切入，再进入规则或行业逻辑。

标题和小标题要求：

- 禁止连续复用“不是 X，是 Y”“X 变了，Y 先感受到”“这次变化说明 X”等固定句式。
- 禁止把通用结构标签当小标题，如“事实底座”“背景解释”“原创分析”“普通人影响”“中国变量”“结语”。
- 小标题必须是内容专属句，能让读者不看正文也知道这一节的新信息。

## 视觉规则改造

封面从单模板改为封面原型轮换：

- 纯标题排版型：仅作为备用或手动快速封面，不作为 cron 默认连续使用。
- 数据冲击型：一个核心数字或比例作为视觉中心。
- 结构地图型：行业链路、利益关系、平台规则或技术路线。
- 场景切入型：办公室、通勤、家庭消费、小店、订单、账单等具体场景。
- 对比型：过去/现在、平台/商家、表面/本质、旧规则/新规则。
- 物件隐喻型：合同、价格标签、芯片、订单页、工具界面等。
- 时间线型：几个关键节点组织封面。
- 角色型：抽象角色，不用侵权肖像或真实媒体 logo。

同账号近 7 天不得重复同一个 `cover_archetype + cover_layout + title_pattern`。正文图也按内容选择信息图、场景图、对比图、结构图，不再固定 `baoyu-comic + dramatic/neutral + 1024x576`。

视觉相似性不依赖识图能力。定时任务生成图片前写 `visual-meta.json`，记录封面原型、构图、视觉主体、正文图类型和生图提示词摘要；发布脚本读取这些文本元数据做同账号近期相似性 warning。

## 代码兜底

新增轻量结构签名：

- `structure_archetype`：根据标题、小标题、元素顺序推断文章结构类型。
- `title_pattern`：识别标题句式。
- `opening_pattern`：识别开头方式。
- `heading_signature`：记录 H2/H3 的归一化骨架。
- `element_sequence`：记录表格、列表、引用、图片、代码块等顺序。
- `generic_heading_count`：统计通用模板小标题数量。
- `cover_signature`：记录封面文件、尺寸、扩展名；prompt 侧补充封面原型。

发布前读取 `publish-history.jsonl`，只比较同账号近期记录：

- 同账号结构原型连续重复，告警。
- 小标题骨架高度相似，告警。
- 标题句式重复，告警。
- 通用模板小标题过多，告警。
- 历史中已有封面原型字段且重复，告警。

正式发布成功后把签名写入 `record_publish(extra={...})`，供后续对比。

## 验证

- 更新 unittest 覆盖结构签名生成、同账号相似告警、跨账号不误报。
- 跑 `python3 -m unittest discover -s tests`。
- 手工复查 prompt 是否仍能完成原有写作 → 配图 → dry-run → 发布路径。

## v3.2 轻量化补充

反低创作度规则落地后，cron prompt 不应继续复制完整质量规则。最终形态：

- `SKILL.md` 负责运行契约、硬门禁、命令路径和必须读取的文件。
- `prompts/quality-and-risk.md` 负责质量与风险总则。
- `publish_pipe.py` / `publish_history.py` 负责可代码化的兜底检查。
- `cron-prompts/*.md` 只保留任务差异、执行流程、产物要求、发布命令和报告字段。

这样能降低上下文负担，也减少模型被长篇模板锚定后反复生成相似文章的风险。
