## 问题 1:排版时是否用对了格式(表格/代码块/列表等)

### 现状
`prompts/markdown-elements.md` 已经写得相当完善:
- 13 种元素完整说明 + 渲染效果
- 双账号"必用建议"(2000-4000 字 ≥3 种非段落元素)
- 反例 vs 正例(纯段落堆叠 vs 表格分层)
- 写稿决策流程("这段在做对比→直接写表格")

### 缺口(诊断)
**prompt 充分,但缺乏"结构质量"的事后校验**。`validate_publish_ready` 只校验标题/图片/AI 披露,**没有任何结构维度校验**。结构达标(是否用了表格/列表/代码块、非段落元素占比)完全靠 agent 自觉。

风险:agent 可能交出一篇 80% 纯段落的稿子,没有触发任何拦截。

### 候选方向(正式立项时评估)
1. **纯 prompt 层**:在写作快路径加一段"交付前结构自检 checklist",强制 agent 自报非段落元素数量。零代码,但靠自觉。
2. **代码层 warning 扫描**:新增 `check_structure_quality(md)` 函数,统计非段落元素(表格/列表/引用/代码块)数量,低于阈值(如 <3 种)打 warning,接入 `validate_publish_ready` 的 warning 通道(不阻断,和风险关键词同级)。更可靠。
3. **两者结合**:prompt 教 agent 自检 + 代码做兜底扫描。

倾向方向 2/3。warning 级而非 block,避免误伤短文。

## 问题 2:baoyu 配图在正文注入时的断层

### 现状
配图生成流程:读 `baoyu-style-index.md` 选风格 → 构造英文 prompt(中文 ≤2 主题词) → 调 Agnes API → 下载图片 → 在 markdown 写 `<!-- img:filename -->` → 跑 `publish_pipe.py --images <路径>`。

### 缺口(诊断)

**A. 风格/数量规则的强制力不足**:
- `baoyu-style-index.md` 5.1 有配图数量硬规则(<2500字 0-1张,≥2500字 0-2张)、5.2 四类触发条件、5.3 禁止插图场景。
- 但这些规则**没有任何代码或检查点强制**,完全靠 agent 读 prompt 时记得并遵守。
- 风险:agent 可能给短文配 3 张图(超标),或给纯观点段落配图(5.3 禁止场景)。

**B. marker 与图片文件名交接断层(更关键)**:
- 没有机制保证 `<!-- img:hero.png -->` 里的文件名 == agent 实际下载的图片文件名。
- 现状是事后发现:`publish_pipe.py --dry-run` 才报"找不到图片"。**浪费一次图片生成**(Agnes 调用 + 420s 超时)。
- 应该是事前预防:先定图位命名,再生成,注入时校验一致。

### 候选方向(正式立项时评估)
1. **注入流程固化**:在 SKILL.md 配图快路径里,规定固定步骤——①先确定图位和命名(`hero.png`/`flow.png`)→ ②生成时 `--out` 用同名绝对路径 → ③marker 用同名 → ④dry-run 校验。纯流程,无代码。
2. **配图数量 warning 扫描**:新增 `check_image_count(md, image_files)`,对比正文字数与 baoyu 索引 5.1 的数量阈值,超标打 warning。
3. **marker ↔ 文件一致性预检**:在生成图片前(调 Agnes API 前),先扫 markdown 里所有 `<!-- img:xxx -->`,列出预期的文件名清单,让 agent 确认每个都有对应文件再生成。事前预防。

倾向 1 + 3 组合。1 固定流程,3 提供一致性预检点。
