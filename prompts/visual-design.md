# 视觉与标题规范（公众号图文必读）

## 1. 字体（强制）

所有 HTML 图（封面 cover、正文图 hero / 模块图）默认字体栈：

```
"XinYuGongHeXieSong", "WQYZenHeiLocal", "WenQuanYi Zen Hei",
"Noto Sans CJK SC", "PingFang SC", "Microsoft YaHei", sans-serif
```

字体已注册在 `/home/ubuntu/.fonts/xinyugong/`，三套模板
（`editorial-cover-v2.html` / `editorial-body-v1.html` / `editorial-body-modular-v1.html`）
都已内嵌 `@font-face`。新增模板必须沿用同一字体栈，不要回退到系统默认。

字重对照：
- 300 Light（副标题/小字注解）
- 400 Regular（正文）
- 500 Medium（卡片标题）
- 600 SemiBold（栏目/分区标题）
- 700 Bold（主标题）
- 900 Heavy（封面主视觉大字 / 引爆字）

封面主视觉大字优先 700~900，正文不超过 500，避免整图过黑。

## 2. 标题与封面主视觉文案（悬念感）

公众号点击率 80% 决定于标题与封面文案。这个 skill 下生成标题时**强制遵守**以下规则：

### 必须做到
- **抛出张力或问题，不给结论**。读者要"想知道答案"才会点。
- **具体名词冲突 / 数字反差**优先于抽象概念。
- 信息密度 ≤ 60%，剩下 40% 留给好奇心。
- 主标题 ≤ 18 字；封面大字 ≤ 12 字（最好 6–10 字 + 一行副线索）。
- 副标题/副线索可以补充背景，但不能直接给出答案。

### 禁止
- ⛔ 把结论写进标题。例："Hermes 完胜 OpenClaw" → 改为"Hermes 和 OpenClaw，谁先掉队？"
- ⛔ "深度对比 / 全面解读 / 一文看懂"等模板词
- ⛔ "保姆级 / 干货满满 / 强烈推荐"等公众号烂俗 hook
- ⛔ "本文将……" / "今天我们聊聊……"开头
- ⛔ 标题里写 AI / 大模型协助等任何 AI 身份披露（与 publish_pipe AI guard 同步）

### 套路参考（可组合，不要硬抄）
1. **反差式**："xxx 是答案，但问题不是 xxx"
2. **指认式**："真正决定 xxx 的，是被忽视的那一项"
3. **悬念式**："xxx 没说的那部分"
4. **代价式**："xxx 赢了，但代价是什么"
5. **错位式**："看上去是 A，其实是 B"
6. **时间感**："xxx 之前的最后一周"
7. **信息差+震惊体**（泛人群杀手锏）：
   - "刚刚发布，但更在意的是背后那行没人注意的小字"
   - "XX 封禁了 XX，但真相和你想的不一样"
   - 反常识点："都说中国 XX 落后，但这个领域其实已经反超"
   - "你以为 XX 是美国的优势，其实那是中国的主场"

写完标题先自检：把标题给一个完全没看过文章的人，TA 会不会想"啊？为什么？" —— 不会就重写。

## 3. 封面图（cover）设计原则

模板：`editorial-cover-v2.html`，900×383 横版，编辑风。

- **主标题大字优先视觉锤**，颜色对比强烈（navy / red / 金）；不要全黑大字。
- **副标题压在主标题下，颜色 muted**，密度低 1/3，给主标题"留白"。
- 留 1 处"视觉钩子"：可以是一个数字、一个红印、一个英文短词、一个箭头。**不要塞满**。
- 不要在封面写超过 2 行正文级别的话，封面是"招牌"，不是"摘要"。
- 颜色调性沿用 paper（#f6f1e8）/ navy / red / 金，避免亮蓝亮绿亮粉等浮夸色。

## 4. 正文图（hero / 模块图）设计原则

**核心理念：图应该是"省读者脑力"的工具，不是"装饰文章"的道具。**

文字能讲清的，绝对不画。读者滑到正文图，应该是"啊，这下我看明白了"，而不是"嗯，又一张装饰"。

### 配图数量（硬规则）

| 正文字数 | 配图数 |
|---|---|
| < 2500 字 | 0–1 张 |
| ≥ 2500 字 | 0–2 张 |

**数量永远可以是 0。**不要为了"长文必须配图"凑数。

### 何时该插图（四类触发条件，满足任一才插）

1. **数据 / 数字** → 一个核心数字比一段文字更震撼。
   *例：调用量从 12% 涨到 67% → 一张大字数字 hero 图*
2. **对比** → A vs B 的并列结构。
   *例：旧做法 vs 新做法 / 路线 A vs 路线 B / 表面 vs 实际*
3. **流程 / 演化** → 有时间或步骤的走向。
   *例：阶段 1 → 阶段 2 → 阶段 3 / 决策树 / 转型路径*
4. **结构关系** → 文字说不清的拓扑或分层。
   *例：三层抽象 / 四象限 / 系统组件关系*

### 何时不该插（明确禁止）

- ⛔ 纯观点段落 —— 再精彩也别配图，破坏阅读节奏
- ⛔ 单一概念定义 —— 一句话能说清的，不要画
- ⛔ 列表已经能说清的 —— 已有 1-2-3-4 编号就别再画卡片图
- ⛔ "为了让文章看起来更专业" —— 这本身就是要打掉的冲动

### 设计原则

- 模板优先用 `editorial-body-v1.html`（单图 hero）；只有真的需要"结构展开"时才上 `editorial-body-modular-v1.html`
- 信息密度**低**比高重要：一张图聚焦一个判断，多一个元素就要砍一个
- 颜色与封面一致，保持系列识别度
- 不要在图里复述正文段落的整句话——要么提炼成名词，要么改成数字

### 正文图的"装饰预算"——最多 2 个视觉元素

正文图不是封面、不是杂志页、不是 PPT。**每张图只服务一个判断**。

按这个清单 **逐项删**，删到只剩下两个为止：

| 元素 | 默认 | 何时保留 |
|---|---|---|
| 顶部 kicker / 栏目标签（"DATA · XXX"）| **删** | 不保留 |
| 主标题（一句话讲清这张图想说什么） | **必留** | 必留 |
| 主图形（柱/条/对比块/流程节点） | **必留** | 必留 |
| 数值标注（k / % / 数字） | 留 | 必留 |
| 副标注（"2025-07 起步" 之类的上下文）| **删** | 唯独此信息不放图里就无法读懂主图时 |
| 底部反转金句 / 编者按 | **删** | 写进正文段落，不进图 |
| 数据来源 / 截止日期 | **删** | 写进正文图后面的小字注脚段落，不进图 |
| 装饰线 / 分割条 / 角标 | **删** | 不保留 |

判断标准：**遮住任何一个元素，主判断还能读懂吗？能 → 该元素删掉。**

理由：

- 反转金句、数据来源、kicker 都是"文章的事"，让它们留在正文段落里，读者读完图自然衔接到下一段，节奏更顺
- 图里的字越少，那个核心数字 / 对比关系就越炸
- 装饰会让 agent 误以为"图越满越专业"，反而把每张图都做成迷你海报

### 设计图前的自问

> "如果我把这张图换成一句话写在正文里，读者会不会更累？"
>
> - 不会更累 → 这张图不该存在
> - 会更累，但累在"数据需要逐个对比" → 该画，且只画对比本身
> - 会更累，但累在"流程需要顺序感" → 该画，且只画箭头链

agent 不要从"我有什么素材"出发设计图，要从"读者哪一刻会读不下去"出发。

## 5. 检查清单（生成图前必读）

- [ ] 标题给的是"勾"还是"结论"？是结论 → 重写
- [ ] 封面大字 ≤ 12 字？
- [ ] 字体栈第一项是 XinYuGongHeXieSong？
- [ ] **这张正文图是数据/对比/流程/结构 中的哪一类？** 都不是 → 不要插
- [ ] 配图数量符合规则？（<2500 字 0–1 张，≥2500 字 0–2 张）
- [ ] **这张图只剩 2 个视觉元素了吗？**（主标题 + 主图形 + 数值，其它都该砍掉）
- [ ] 颜色是否在 paper / navy / red / 金 范围内？
- [ ] 有没有任何 AI 身份披露文字混进图里？
