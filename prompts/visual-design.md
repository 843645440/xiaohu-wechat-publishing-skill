# 视觉与标题规范（公众号图文必读）

## 1. 字体（强制）

所有 HTML 图（封面 cover、正文图 hero / 模块图）默认字体栈：

```
"XinYuGongHeXieSong", "WQYZenHeiLocal", "WenQuanYi Zen Hei",
"Noto Sans CJK SC", "PingFang SC", "Microsoft YaHei", sans-serif
```

字体已注册在 `/home/ubuntu/.fonts/xinyugong/`，三套模板
（`editorial-cover-v2.html` / `editorial-body-v1.html` / `editorial-body-modular-v1.html`）
都已内嵌 `@font-face`。新增模板必须沿用同一字体栈，不要回退到系统默认。

字重对照：
- 300 Light（副标题/小字注解）
- 400 Regular（正文）
- 500 Medium（卡片标题）
- 600 SemiBold（栏目/分区标题）
- 700 Bold（主标题）
- 900 Heavy（封面主视觉大字 / 引爆字）

封面主视觉大字优先 700~900，正文不超过 500，避免整图过黑。

## 2. 标题与封面主视觉文案

> 服从 `prompts/quality-and-risk.md`。标题/封面文案是营销号判定的高发区，标题党与夸大承诺会直接拖累推流。

标题和封面文案要**准确反映正文、可带观点、有适度悬念，但不夸大、不贩卖焦虑、不做不兑现的承诺**。

### 先记一条总原则（这次踩坑后的强制修正）
- **标题不是每次都要做悬念型。** 要根据题材和场景选标题风格，而不是默认用“谁在买单 / 还是证据吗 / 谁在补差价”这类抽象钩子。
- **标题第一眼必须看懂文章在讲什么。** 至少保留 1 个明确主体词：公司、产品、平台、行业、场景、品类之一。
- 如果读者看完标题还不知道你在讲苹果、外卖、补贴、AI 相册、平台规则变化，那就是坏标题，直接重写。
- 悬念可以有，但悬念必须建立在**主体明确**的前提上。

### 这次真实踩坑（以后不要再犯）
以下两类标题已证明有问题：

1. **主体缺失型**
   - 反例：`当 AI 开始改你的相册，照片还是证据吗？`
   - 问题：正文其实在讲**苹果相册 / 苹果 AI 修图功能**，但标题里没有“苹果”这个主体。读者第一眼不知道你在讲哪家公司、哪件具体事。

2. **场景缺失型**
   - 反例：`便宜还在，谁在补那笔差价？`
   - 问题：正文其实在讲**外卖补贴 / 平台与商家成本分摊**，但标题里没有“外卖 / 平台 / 补贴”这些关键词。读者不知道“什么便宜、什么差价”。

**结论：** 标题不能只剩下情绪钩子，必须保留题目对象。否则用户看不懂在讲什么，就不会点。

### 必须做到
- **抛出真实的张力或问题，但答案必须在正文里兑现**。悬念来自“这事我没想到”，不是“标题骗你点进来”。
- **具体名词 / 真实数字**优先于抽象概念；数字必须有来源、不夸大。
- **主标题优先清楚，再考虑悬念。** 看不懂的悬念标题一律不如看得懂的信息标题。
- 主标题 ≤ 18 字；封面大字 ≤ 12 字（最好 6–10 字 + 一行副线索）。
- 副标题/副线索补充背景，可留悬念，但不能误导。
- 风险软规避同样适用于标题/封面：**不预测涨跌、不评判政策、不碰敏感时政地缘**（详见 `quality-and-risk.md` C 节）。

### 禁止
- ⛔ 标题党 / 空震惊：标题喊“真相和你想的不一样”，正文却没有实质反转。
- ⛔ 无证据的颠覆式断言、夸张承诺、贩卖焦虑。
- ⛔ 抽象悬念大于具体信息：看完标题还不知道文章在讲什么事。
- ⛔ “深度对比 / 全面解读 / 一文看懂”等模板词。
- ⛔ “保姆级 / 干货满满 / 强烈推荐”等烂俗 hook。
- ⛔ “本文将……” / “今天我们聊聊……”开头。
- ⛔ 标题里写 AI / 大模型协助等任何 AI 身份披露（与 publish_pipe AI guard 同步）。
- ⛔ 标题/封面命中高风险词（涨跌预测、政策评判、敏感时政）——会触发软扫描告警，应换软切口。

### 标题风格库（按场景选，不要每次都悬念型）

#### 1. 信息直给型
适合：新闻性强、事件本身够新、需要让人一眼看懂。

写法：
- `苹果把 AI 修图放进相册后，照片还算证据吗？`
- `外卖补贴开始被规范后，低价还能维持多久？`

要求：主体清楚、事件清楚、问题清楚。

#### 2. 问题讨论型
适合：正文围绕一个清晰问题展开。

写法：
- `苹果把 AI 修图塞进相册后，照片还能信吗？`
- `平台补贴还在，外卖这笔差价到底谁在扛？`

要求：问题必须具体，不能悬空。

#### 3. 观点判断型
适合：正文有明确核心判断、作者立场比较强。

写法：
- `苹果正在把相册，从记录工具变成生成工具`
- `外卖最危险的不是补贴少了，而是好店会先消失`

要求：标题里的判断，正文必须撑得住。

#### 4. 影响切口型
适合：强调“这和普通人有什么关系”。

写法：
- `当苹果把 AI 修图做进相册，普通人该怎么留证据？`
- `外卖补贴规则变了，最后影响的不是红包，是你常点的小店`

要求：把“普通人影响”说清楚，不要空喊。

#### 5. 反差式
适合：有“表面 A，其实 B”的结构。

写法：
- `苹果看上去在升级相册，其实改的是照片的真实性`
- `看上去是外卖更便宜了，其实平台在重算谁来买单`

要求：反差要真实，不能硬拗。

#### 6. 场景代入型
适合：生活化强、适合公众号读者代入。

写法：
- `以后用苹果手机拍下来的照片，你还敢直接当证据吗？`
- `你点到的 9 块 9 外卖，最后可能不是平台在补`

要求：场景必须具体，不能虚。

#### 7. 解释型
适合：想稳一点、清楚一点，不追求过强情绪。

写法：
- `为什么苹果把 AI 放进相册，会让照片变得没那么可靠？`
- `为什么外卖越便宜，商家反而越难做？`

要求：适合深度文、解释文、信息量高的稿子。

### 套路参考（可组合，不要硬抄；都必须由正文兑现）
1. **指认式**："真正决定 xxx 的，是被忽视的那一项"
2. **悬念式**："xxx 没说的那部分"
3. **代价式**："xxx 之后，普通人多付的那笔钱"
4. **错位式**："看上去是 A，其实是 B"
5. **时间感**："xxx 之前的最后一周"
6. **影响式（软规避优先）**："xxx 变了，你的工资/房贷/账单会怎么动"

写完标题先自检三条：
- 看完标题，外行人能不能立刻说出“这篇在讲什么事 / 什么对象”？不能就重写。
- 把标题给一个没看过文章的人，TA 会想“啊？为什么？”——不会就重写。
- 正文**真的兑现**了标题抛出的那个悬念或判断吗？没兑现就是标题党，改标题或补正文。

## 3. 封面图（cover）设计原则

模板：`editorial-cover-v2.html`，900×383 横版，编辑风。

- **主标题大字优先视觉锤**，颜色对比强烈（navy / red / 金）；不要全黑大字。
- **副标题压在主标题下，颜色 muted**，密度低 1/3，给主标题"留白"。
- 留 1 处"视觉钩子"：可以是一个数字、一个红印、一个英文短词、一个箭头。**不要塞满**。
- 不要在封面写超过 2 行正文级别的话，封面是"招牌"，不是"摘要"。
- 颜色调性沿用 paper（#f6f1e8）/ navy / red / 金，避免亮蓝亮绿亮粉等浮夸色。

## 4. 正文图（hero / 模块图）设计原则

**核心理念：图应该是"省读者脑力"的工具，不是"装饰文章"的道具。**

文字能讲清的，绝对不画。读者滑到正文图，应该是"啊，这下我看明白了"，而不是"嗯，又一张装饰"。

### 正文图来源：先选类型，再选工具

正文图有两条路，按内容性质二选一，不要混用：

- **默认 = baoyu-comic 知识漫画**（Agnes API 生成，manga + dramatic/neutral）。叙事、场景、概念、情绪类内容首选漫画——用画面讲故事。扁平信息图被认为"廉价"，非数据类不要用。
- **HTML 信息图**（`editorial-body-v1.html` / `editorial-body-modular-v1.html` 渲染）。**只在下面"何时该插图"的数据/对比/流程/结构四类**确实需要精确呈现时才用。

> 判断顺序：先问"这张图要呈现的是精确的数据关系吗？"——是 → HTML 信息图；否（是叙事/概念/场景）→ baoyu-comic 漫画。本节下面的"装饰预算/元素清单"针对 **HTML 信息图**；漫画风的提示词规范见 `references/body-image-baoyu-agnes-workflow.md`。

### 配图数量（硬规则）

| 正文字数 | 配图数 |
|---|---|
| < 2500 字 | 0–1 张 |
| ≥ 2500 字 | 0–2 张 |

**数量永远可以是 0。**不要为了"长文必须配图"凑数。

### 何时该插图（四类触发条件，满足任一才插）

1. **数据 / 数字** → 一个核心数字比一段文字更震撼。
   *例：调用量从 12% 涨到 67% → 一张大字数字 hero 图*
2. **对比** → A vs B 的并列结构。
   *例：旧做法 vs 新做法 / 路线 A vs 路线 B / 表面 vs 实际*
3. **流程 / 演化** → 有时间或步骤的走向。
   *例：阶段 1 → 阶段 2 → 阶段 3 / 决策树 / 转型路径*
4. **结构关系** → 文字说不清的拓扑或分层。
   *例：三层抽象 / 四象限 / 系统组件关系*

### 何时不该插（明确禁止）

- ⛔ 纯观点段落 —— 再精彩也别配图，破坏阅读节奏
- ⛔ 单一概念定义 —— 一句话能说清的，不要画
- ⛔ 列表已经能说清的 —— 已有 1-2-3-4 编号就别再画卡片图
- ⛔ "为了让文章看起来更专业" —— 这本身就是要打掉的冲动

### 设计原则

- 模板优先用 `editorial-body-v1.html`（单图 hero）；只有真的需要"结构展开"时才上 `editorial-body-modular-v1.html`
- 信息密度**低**比高重要：一张图聚焦一个判断，多一个元素就要砍一个
- 颜色与封面一致，保持系列识别度
- 不要在图里复述正文段落的整句话——要么提炼成名词，要么改成数字

### 正文图的"装饰预算"——最多 2 个视觉元素

正文图不是封面、不是杂志页、不是 PPT。**每张图只服务一个判断**。

按这个清单 **逐项删**，删到只剩下两个为止：

| 元素 | 默认 | 何时保留 |
|---|---|---|
| 顶部 kicker / 栏目标签（"DATA · XXX"）| **删** | 不保留 |
| 主标题（一句话讲清这张图想说什么） | **必留** | 必留 |
| 主图形（柱/条/对比块/流程节点） | **必留** | 必留 |
| 数值标注（k / % / 数字） | 留 | 必留 |
| 副标注（"2025-07 起步" 之类的上下文）| **删** | 唯独此信息不放图里就无法读懂主图时 |
| 底部反转金句 / 编者按 | **删** | 写进正文段落，不进图 |
| 数据来源 / 截止日期 | **删** | 写进正文图后面的小字注脚段落，不进图 |
| 装饰线 / 分割条 / 角标 | **删** | 不保留 |

判断标准：**遮住任何一个元素，主判断还能读懂吗？能 → 该元素删掉。**

理由：

- 反转金句、数据来源、kicker 都是"文章的事"，让它们留在正文段落里，读者读完图自然衔接到下一段，节奏更顺
- 图里的字越少，那个核心数字 / 对比关系就越炸
- 装饰会让 agent 误以为"图越满越专业"，反而把每张图都做成迷你海报

### 设计图前的自问

> "如果我把这张图换成一句话写在正文里，读者会不会更累？"
>
> - 不会更累 → 这张图不该存在
> - 会更累，但累在"数据需要逐个对比" → 该画，且只画对比本身
> - 会更累，但累在"流程需要顺序感" → 该画，且只画箭头链

agent 不要从"我有什么素材"出发设计图，要从"读者哪一刻会读不下去"出发。

- [ ] 配图数量符合规则？（<2500 字 0–1 张，≥2500 字 0–2 张）
